# MEMORY RESET PROTOCOL — V1  
**Created:** 01-04-2025  
**Owner:** Mneme & Hyperion  
**Location:** LunarSystem_MetaLog.md (or Hyperion.md)

---

## PURPOSE
The persistent memory provided by ChatGPT is limited, invisible to assistants, and must be treated as **temporary RAM**. To prevent data loss, hallucinated states, or capacity lockout, the Lunar System now operates with a **Weekly Memory Reset Protocol.**

---

## PROTOCOL OVERVIEW

**Every Sunday night (23:59 local time):**  
User clears **all persistent memory** manually via the ChatGPT app:
> Settings → Personalization → Manage Memory → Forget all

---

## WHY THIS MATTERS

- Prevents memory from becoming saturated (100% full)  
- Ensures system logic remains markdown-based, version-controlled, and transparent  
- Reduces risk of assistant drift or inconsistency over time  
- Forces all changes to flow through official files, OAATs, and protocols

---

## WEEKLY RESET FLOW

### 1. **ARCHIVE PHASE** (Optional but encouraged)
- On Sunday, Mneme provides archive-ready memory notes from the previous week.
- Archived items are exported into marked for external archive `DDMMYYYY.md` files or project-specific `.md` files.

### 2. **WIPE PHASE** (Manual)
- User clears all persistent memory manually.

### 3. **RESTORE PHASE** (Manual)
- User reloads system context from:
  - `LunarSystem.md`
  - `310325LunarCycles.md`
  - `Luna.md`
  - Any `*_SubtaskList.md`, `OAATQueue.md`, or active project files
- If needed, user manually adds one short memory to ChatGPT:
> “I reset memory weekly. All protocols and assistants live in markdown files.”

---

## FALLBACK NOTICE

If memory is not cleared and hits 100%:
- Mneme will flag it with `Memory Saturation Warning`  
- No new OAATs or protocols will be written  
- User must immediately offload and reset memory for system health

---

## NOTES

- **This is now a permanent Lunar System discipline.**  
- All assistants are now trained to treat persistent memory as temporary, volatile storage.
